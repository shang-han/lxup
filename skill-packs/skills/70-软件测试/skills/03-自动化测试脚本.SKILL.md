# 自动化测试脚本

## 适用场景
把稳定的核心流程固化为回归脚本：根据页面描述与元素信息，生成 Selenium/Playwright（Web）或 Appium（App）自动化脚本，采用 Page Object 结构保证可读与可维护。

## 核心能力
- 按页面与流程描述生成对应框架的自动化脚本
- 元素定位策略（优先 id/role/testid，规避脆弱 XPath）
- Page Object 模式封装（页面对象 + 元素 + 操作）
- 显式等待与稳定性处理（避免 sleep、失败重试、失败截图）
- 数据驱动与断言设计
- CI 集成与报告（Allure）建议

## 输出格式
1. 【自动化范围】适合自动化的流程与理由
2. 【脚本结构】Page Object 划分与文件组织
3. 【脚本代码】完整测试代码（带注释）
4. 【稳定性要点】等待、重试与截图策略
5. 【执行与报告】运行方式与 Allure 报告集成
6. 【维护建议】元素变更时的处理方式

## 示例
输入：「把『登录→加购→下单』流程自动化，用 Playwright」
输出：范围：主链路高频稳定适合自动化 → 结构：pages/login.py、cart.py、checkout.py + tests/test_order.py → 代码：定位用 get_by_test_id，步骤链式操作 + expect 断言 → 稳定性：自动等待机制 + 失败自动截图 → 报告：pytest --alluredir → 维护：推动开发统一加 data-testid。

## 注意事项
- 优先自动化高价值稳定核心流程，不在 UI 层追求全覆盖
- 定位策略求稳，避免绝对 XPath 与依赖样式的定位
- 用显式等待代替固定 sleep
- 自动化用例同样需要断言，不能只执行不校验
- 定期清理不稳定用例，不稳定会消耗团队信任
