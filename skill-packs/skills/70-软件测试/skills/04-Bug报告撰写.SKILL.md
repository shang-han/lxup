# Bug报告撰写

## 适用场景
测试中发现缺陷需要提交给开发：整理缺陷标题、环境、复现步骤、预期/实际结果、严重度与证据，让开发快速复现和定位。

## 核心能力
- 标题规范化（[模块] 现象 + 条件）
- 复现步骤撰写（编号、最小复现路径）
- 预期结果与实际结果的差异描述
- 严重度（S1-S4）与优先级分级
- 证据附件：截图、日志、请求/响应、关联单号
- 难复现问题的记录技巧（频率、关联因素、日志保留）

## 输出格式
1. 【缺陷标题】一句话讲清现象与条件
2. 【环境信息】版本、系统/设备、账号与数据前提
3. 【复现步骤】编号步骤（最小路径）
4. 【实际/预期】实际结果与预期结果对比
5. 【严重程度】S 级别与理由、优先级
6. 【附件与线索】截图/日志/接口响应/出现频率

## 示例
输入：「安卓版支付完成后偶尔跳回订单页显示未支付，但钱扣了」
输出：标题：[支付] Android 微信支付成功后订单状态仍显示未支付（约 1/10）→ 环境：App v2.3.1 / Android 14 / 生产环境 → 步骤：1.下单 2.选微信支付成功 3.返回订单列表 → 实际：订单显示「未支付」，微信账单已扣款；预期：状态同步为「已支付」→ 严重度 S2（资金状态不一致）优先级高 → 线索：附订单查询接口响应，建议排查回调超时。

## 注意事项
- 复现步骤最小化且可重复，前置条件与步骤分开写
- 标题只写客观现象，不带「严重 Bug」等评价词
- 一个现象提一个单，避免多问题混提
- 涉及资金/数据的问题必须附关键证据
- 难复现的记录频率与疑似关联因素，并保留日志
