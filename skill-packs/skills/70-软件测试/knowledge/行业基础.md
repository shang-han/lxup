# 行业基础

## 行业概况
软件测试工程师是产品质量的守门人，负责需求评审、用例设计、功能/接口/自动化/性能测试与缺陷跟踪。行业正从手工测试转向测试开发：接口自动化与 CI/CD 集成是标配，质量保障向左移（需求与设计阶段）和右移（线上监控）延伸。优秀的测试不只发现 Bug，更能预防 Bug、推动质量体系化建设。

## 专业技能体系
### 硬技能
- 用例设计方法（等价类/边界值/场景法/正交表）
- 接口测试（Postman/Jmeter/Pytest）
- 自动化测试框架（Selenium/Appium/Playwright）
- 缺陷管理与追踪
- 性能测试基础（压测/并发/瓶颈定位）
### 软技能
- Bug描述与重现步骤撰写
- 开发沟通与推进修复

## 常用术语
- 用例：测试用例，含步骤与预期结果
- 冒烟测试：版本提测后对主流程的快速验证
- 回归测试：变更后对已有功能的再验证
- 等价类/边界值：划分有效/无效区间与边界点设计用例的方法
- TPS/QPS：每秒处理的事务数/查询数
- 复现率：缺陷能被重现的概率
- 缺陷生命周期：新建→指派→修复→验证→关闭的状态流转
- 测试左移：质量活动前置到需求与设计阶段

## 核心工作流程
需求阶段：需求评审 → 识别测试点与风险 → 用例设计 → 用例评审
执行阶段：冒烟测试 → 功能/接口测试 → 缺陷提交与跟踪 → 回归测试
专项：自动化脚本 → 接入流水线 → 定期回归；性能压测
发布阶段：发布标准检查 → 测试报告 → 线上验证
持续：缺陷复盘 → 用例补充 → 流程改进

## 常见问题
Q：开发说「无法复现」怎么办？
A：提供最小复现步骤与环境数据，附日志/截图/请求 ID；当面演示；记录频率与关联因素（账号、数据、时间），保留日志供排查。

Q：自动化与手工测试怎么分配？
A：高频稳定的核心回归适合自动化；新功能、一次性场景与体验类测试用手工。自动化追求投入产出比，不追求覆盖率数字。

Q：如何推动开发修复不愿修的缺陷？
A：用事实与数据说话（影响范围、用户损失、复现率），明确严重度分级；必要时升级到项目经理或产品按风险决策。

Q：怎么保证测试覆盖完整？
A：多方法组合：需求点覆盖矩阵、用例设计方法（等价类/边界值/场景法）、基于历史缺陷的错误推测、用例评审互查。
