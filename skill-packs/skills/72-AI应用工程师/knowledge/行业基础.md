# 行业基础

## 行业概况
AI 应用工程师是把大模型能力落到业务场景的角色，位于基础模型与业务需求之间。随着大模型 API（OpenAI/DeepSeek/通义千问）与开发框架（LangChain/Dify/Coze）普及，应用开发门槛大幅降低，竞争焦点转向效果优化、工程化落地与成本控制。行业格局：基础模型厂商提供能力，应用层靠场景理解、私有数据与体验取胜，工程师的核心价值在于后两者的结合。

## 专业技能体系
### 硬技能
- 大模型API调用与Prompt工程（OpenAI/DeepSeek/通义千问）
- RAG系统搭建（LangChain/Milvus/Chroma）
- Agent与多智能体编排（LangGraph/Coze/Dify）
- 模型微调基础（LoRA/QLoRA/SFT）
- 向量数据库与混合检索优化
### 软技能
- AI能力边界判断
- 业务需求→AI方案转化

## 常用术语
- Prompt工程：通过设计与优化输入指令引导模型输出的方法体系
- RAG（检索增强生成）：生成前先检索外部知识库再作答，缓解幻觉、支持溯源
- Embedding：将文本转为向量表示，是语义检索的基础
- 向量数据库：专门存储向量并做相似度检索的数据库（Milvus/Chroma/FAISS）
- Function Calling：大模型调用外部工具/API 的机制，Agent 的技术基础
- Agent（智能体）：能自主规划、调用工具、多步完成任务的 AI 系统
- LoRA/QLoRA：低秩微调方法，只训练少量参数即可定制模型，显存开销小
- Token：模型处理文本的最小单位，计费与上下文长度的依据

## 核心工作流程
需求阶段：业务需求调研 → AI 可行性判断 → 方案设计与选型
实现阶段：数据准备 → 原型开发（Prompt/RAG/Agent）→ 效果评估与调参
交付阶段：POC 验收 → 成本与性能优化 → 上线监控（效果漂移、成本、内容安全）
迭代阶段：badcase 收集 → Prompt/数据/模型迭代 → 版本评估与灰度

## 常见问题
Q：什么时候用 RAG，什么时候用微调？
A：知识时效性与溯源问题用 RAG（知识会变、答案要引用来源）；说话风格、输出格式、领域能力问题用微调。企业场景 80% 应先把 RAG + Prompt 做到极致，再评估微调的边际收益。

Q：怎么降低大模型 API 成本？
A：模型分层（简单任务路由到小模型）、高频问题缓存、压缩 Prompt 与控制上下文长度、非实时任务走批量接口。前提是建好评估集，降本不降效果。

Q：RAG 答非所问、编造内容怎么治？
A：分层治理——检索侧用混合检索 + 重排提高命中率，查询侧做查询改写；生成侧 Prompt 约束「仅依据检索内容作答、不知道就说不知道」并要求引用来源；评估侧建测试集量化跟踪，禁止凭感觉调参。

Q：AI 应用上线有哪些合规坑？
A：个人信息进入模型须有授权与必要性评估；生成式服务对外提供须关注算法备案与内容安全审核；数据出境受限；训练语料版权要清晰。方案阶段就把合规写进架构，别上线后补救。
