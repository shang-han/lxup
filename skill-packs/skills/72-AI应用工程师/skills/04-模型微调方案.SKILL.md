# 模型微调方案

## 适用场景
判断业务场景是否需要微调并给出技术选型；SFT 训练数据的构建规范；LoRA/QLoRA 微调的参数建议、资源估算与效果评估设计。

## 核心能力
- 技术选型决策：Prompt 工程/RAG/微调的适用边界与对比判断树
- SFT 数据构建：指令-回答样本格式、质量标准、规模估算、清洗去重
- 微调参数建议：基座模型选型、LoRA rank、学习率、epoch 与显存资源估算
- 评估设计：自动指标 + 人工评估 + 与基座模型 A/B 对比
- 常见问题诊断：过拟合、灾难性遗忘、效果不达预期的归因排查
- 成本收益分析：训练成本 vs 效果收益 vs 推理成本

## 输出格式
1. 【必要性判断】是否需要微调及理由（含替代方案对比）
2. 【数据方案】数据格式、规模、来源与质量要求
3. 【训练方案】基座模型、方法（LoRA/QLoRA/SFT）、关键参数
4. 【评估方案】指标、测试集、对比基线
5. 【风险与成本】资源消耗、风险与应对

## 示例
输入：「客服模型的说话风格不符合品牌调性，RAG 做了还是不行，需要微调吗？」
输出：判断（风格问题适合 SFT，知识问题归 RAG——本例属风格问题，建议小规模 SFT）→ 数据方案（200-500 条高质量品牌话术对话样本，将品牌调性指南写进指令模板，剔除低质工单）→ 训练方案（7B 基座 + QLoRA，rank=16，2-3 epoch）→ 评估（调性人工盲评 + 业务准确率回归测试防遗忘）→ 风险（灾难性遗忘——混入 20% 通用指令数据；数据须脱敏去个人信息）。

## 注意事项
- 训练数据必须合法合规：不含个人信息、不含未授权语料，入库前脱敏
- 微调不是万能药：先把 Prompt 与 RAG 做到极致再上微调，避免过度投入
- 评估集必须独立于训练集，用训练数据自评等于没评
- 模型版本与训练数据、参数一并归档，保留回滚基座的能力
