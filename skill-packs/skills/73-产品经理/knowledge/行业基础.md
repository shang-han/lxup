# 行业基础

## 行业概况
产品经理是互联网产品团队的枢纽，负责把用户需求与商业目标转化为可落地的产品方案，并推动设计、研发、测试、运营协同交付。行业从流量红利期进入存量经营期，对产品经理的要求从「功能实现」升级为「数据驱动增长 + 体验打磨 + 商业化平衡」。方向分化明显：C 端重体验与增长，B 端重业务流程与效率，平台型重生态与规则设计。

## 专业技能体系
### 硬技能
- 需求分析与PRD撰写
- 原型设计（Axure/Figma）
- 用户研究与可用性测试
- 数据驱动决策（埋点/漏斗/留存分析）
- 项目推进与版本排期
### 软技能
- 多方利益平衡与优先级判断
- 用户共情与场景抽象

## 常用术语
- PRD：产品需求文档，研发与测试的核心依据
- MVP：最小可行产品，用最小闭环验证核心假设
- 北极星指标：最能反映产品核心价值交付的单一关键指标
- 漏斗分析：统计用户行为路径各环节的转化率与流失率
- 留存率：新增用户在一段时间后继续使用的比例（次日/7日/30日留存）
- A/B测试：随机分流对比两个版本效果的实验方法
- 埋点：在关键行为位置部署数据采集代码
- RICE：优先级评分法（触达 × 影响 × 信心 ÷ 工作量）

## 核心工作流程
发现：用户研究 → 数据分析 → 需求收集 → 需求池管理
定义：优先级排序 → 版本目标 → PRD 撰写 → 需求评审
设计：原型设计 → 交互与 UI 评审 → 研发测试协同跟进
验证：埋点上线 → 数据复盘 → 用户反馈收集 → 迭代规划

## 常见问题
Q：多方需求冲突时怎么排优先级？
A：回到产品目标与北极星指标，用 RICE 量化各需求的价值与成本，让标准和数据说话而不是比音量；若目标本身冲突，先拉上级对齐目标再排需求。

Q：怎么判断一个需求是伪需求？
A：区分「用户说要什么」和「用户真实问题」：用 5 Whys 挖动机，用数据看行为是否真实存在（类似功能有没有人用），用 MVP 或灰度小成本验证付费/使用意愿。

Q：研发说排期做不完怎么办？
A：不硬压排期：先看范围削减空间（保核心流程、砍次要功能或做简化版），再谈分期交付，最后才申请加人或延期；取舍过程留书面记录，让相关方共同决策。

Q：版本复盘怎么做才有效？
A：结论先行：目标达成度（指标对比）→ 做对了什么/做错了什么 → 原因归因 → 下版本具体行动项。复盘必须产出行动项并有人认领，否则只是流水账。
