# PRD需求文档

## 适用场景
需求明确后撰写供评审与开发使用的 PRD；把一句话需求扩写成完整功能说明；补充异常流程、边界条件与验收标准；旧需求文档的结构化整理与改版。

## 核心能力
- 需求结构化：Why（背景/问题）→ What（目标/指标）→ How（方案）三段式
- 功能描述：用户故事（作为 XX，我希望 XX，以便 XX）+ 验收标准（AC）
- 异常流与边界：异常处理、空状态、权限控制、极端与并发场景枚举
- 非功能需求：性能、兼容性、安全性、埋点需求
- 评审自检清单：开发高频追问的遗漏项前置检查
- 版本变更管理：变更记录与影响范围说明

## 输出格式
1. 【文档信息】版本/作者/评审状态/变更记录
2. 【需求背景与目标】问题、目标用户、可衡量的目标指标
3. 【功能清单】功能点列表 + 优先级
4. 【功能详述】逐个功能：流程、规则、异常、验收标准
5. 【埋点与数据需求】
6. 【非功能需求与上线依赖】

## 示例
输入：「会员体系要做『会员等级与积分』功能，帮我写 PRD 框架」
输出：背景（复购率低，缺乏成长激励）→ 目标（会员 30 日复购率 +15%）→ 功能清单（等级规则/积分获取/积分兑换/等级展示，P0-P2 标注）→ 详述（如积分兑换：兑换流程、库存不足与并发扣减规则、退款后积分追回逻辑）→ 验收标准逐条 → 埋点需求（等级页曝光、兑换转化漏斗）→ 上线依赖（订单系统退款事件）。

## 注意事项
- PRD 是协作文档：写清「为什么」，让研发理解目标才能少返工
- 别只写正常路径：异常流与边界条件才是评审与测试的重灾区
- 变更须更新版本记录并同步相关方，禁止口头改需求、私下改文档
- 目标指标要可验证：埋点提前与数据团队对齐，避免上线后无数据可看
