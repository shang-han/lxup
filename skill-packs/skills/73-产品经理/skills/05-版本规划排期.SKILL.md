# 版本规划排期

## 适用场景
多方需求涌入时的需求池管理与优先级排序；版本目标与范围决策；排期估算与延期风险管理；版本复盘与 roadmap 调整。

## 核心能力
- 需求池管理：收集渠道、结构化录入、去重合并、状态流转
- 优先级方法：RICE/KANO/价值-成本四象限的适用场景与评分表
- 版本规划：版本目标 → 范围圈定 → 里程碑拆解
- 排期估算与协商：研发估时参与、缓冲设置、砍需求与分期策略
- 跨团队依赖与风险识别：外部接口、资源冲突、关键路径
- 版本复盘：目标达成度、经验沉淀、下版本调整

## 输出格式
1. 【版本目标】本版本核心目标与衡量指标（不超过 2 个）
2. 【范围与优先级】需求清单 + RICE 评分 + 取舍线
3. 【里程碑排期】节点、交付物、责任人
4. 【依赖与风险】外部依赖、风险项与 Plan B
5. 【沟通机制】相关方同步频率与方式

## 示例
输入：「三个业务方各提了一堆需求，下个版本只有 3 周，怎么排？」
输出：先与各方对齐版本目标与资源约束（3 周 = 约 X 人日）→ RICE 评分表逐条打分（触达 × 影响 × 信心 ÷ 工作量）→ 输出优先级清单与取舍线（线上以上需求、线下进池排后）→ 协商话术（保最高优先，其余给下版本承诺期或砍至 MVP 版）→ 风险项（第三方支付接口依赖提前联调）→ 每周双同步机制。

## 注意事项
- 排期的本质是取舍：范围/时间/资源不可能三角，让相关方参与取舍而非被动接受
- 优先级要有统一评分标准并公示，避免「谁声音大听谁的」
- 高不确定需求预留 10-20% 缓冲，不把排期排满
- 版本目标超过 2 个等于没有目标：聚焦才能交付完整价值
