# 设计需求分析

## 适用场景
接到产品 PRD 或口头需求后、动手设计前：把业务需求拆解为设计任务，梳理页面信息架构与用户任务流程，明确设计优先级与需求中的模糊点。

## 核心能力
- 从 PRD 中提取用户、场景、目标与核心任务，转化为设计目标
- 页面清单盘点与信息架构（页面层级、跳转关系）
- 核心任务流程拆解（主流程/分支流程/异常流程）
- 识别需求模糊点，生成给产品的待确认清单
- 设计工作量评估与优先级建议
- 竞品参考建议（同类产品中值得借鉴的页面与做法）

## 输出格式
1. 【需求理解】用户、场景、目标、设计目标
2. 【页面清单】页面名称、层级、核心功能
3. 【任务流程】主任务流程步骤与分支情况
4. 【信息架构】页面层级与跳转关系描述
5. 【待确认清单】需与产品确认的问题
6. 【工作量与优先级】设计任务拆解、估时、优先级

## 示例
输入：「要做一个领券中心页，用户可领取和使用优惠券」（附 PRD）
输出：需求理解：目标是提升优惠券领取→使用转化 → 页面清单：领券中心、我的优惠券、用券说明 → 主流程：进入领券中心 → 领取 → 去使用 → 跳转可用商品 → 下单自动选券；分支：已抢光/已过期/不满足门槛 → 待确认 6 条（券能否叠加？领取后有效期从何时算？）→ 工作量：3 页面+5 种状态约 4 人日。

## 注意事项
- 先理解业务目标再拆解设计任务，避免一上来就想页面长什么样
- 异常状态与边界情况要列入清单（空状态、失败、无权限、加载中）
- 待确认清单不能省，口头需求最容易返工
- 信息架构贴合用户心智模型，而非公司组织结构
- 估时预留评审与修改轮次，保守评估
