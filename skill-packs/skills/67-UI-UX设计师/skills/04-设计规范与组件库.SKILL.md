# 设计规范与组件库

## 适用场景
新项目搭建设计规范，或老项目做设计治理：定义设计 Token（颜色/字体/间距/圆角/阴影）、组件规范与使用规则，撰写设计系统文档，保证多页面、多端体验一致。

## 核心能力
- 设计 Token 体系定义（语义化命名、主/辅/中性色、深色模式考虑）
- 字体/行高/间距/栅格规范
- 组件使用规范（按钮/输入框/弹窗/列表/导航：结构、状态、Do/Don't）
- 组件分级与命名建议（与前端组件库命名对齐）
- 响应式多端适配建议（断点、手机/平板/PC 布局规则、安全区）
- 动效规范（时长、缓动曲线、微交互规则）

## 输出格式
1. 【设计原则】3-5 条设计原则
2. 【设计Token】颜色/字体/间距/圆角/阴影规范表
3. 【组件规范】各组件：定义、状态、使用场景、Do/Don't
4. 【栅格与布局】各端断点与布局规则
5. 【动效规范】时长、曲线与应用场景
6. 【落地建议】与前端组件库的协作方式

## 示例
输入：「帮我写一份移动端商城设计规范」
输出：原则：效率优先、反馈清晰、一致可预期 → Token：品牌色 #FF4D4F，语义色 success/warning/error，字号三级 20/16/14，间距基数 4px → 组件：按钮（主/次/文字/禁用四态，点击热区≥44×44）→ 布局：4 列栅格，底部安全区 34px → 动效：标准时长 300ms、ease-out → 落地：Token 映射为 CSS 变量，组件与前端库同名同构。

## 注意事项
- 命名用语义（color-success）而非描述（green），便于换肤
- 组件规范配 Do/Don't 对照示例，比纯文字更有效
- 与前端组件库保持粒度一致，避免设计开发两套体系
- 先建最高频的 20% 组件再迭代，避免过度设计
- 动效统一时长与曲线，避免每页一套规则
