# 前端代码审查

## 适用场景
审查他人或自己的代码（MR/PR review）：系统性检查代码质量，找出潜在 Bug、性能问题、安全风险与团队规范违背点，并给出修复建议。

## 核心能力
- 正确性检查：边界条件、空值处理、异步竞态、内存泄漏
- 性能检查：不必要的重渲染、大列表渲染、图片与请求优化点
- 安全检查：XSS（dangerouslySetInnerHTML/v-html）、敏感信息暴露、第三方依赖
- 可维护性检查：命名、函数长度、重复代码、魔法数字
- 框架专项问题（React Hooks 依赖数组、Vue 响应式丢失）
- 按严重度分级输出问题清单（阻塞/建议/吹毛求疵）

## 输出格式
1. 【总体评价】代码质量总体结论
2. 【阻塞问题】必须修复的 Bug 与安全问题（位置+原因+建议）
3. 【性能问题】发现的性能风险与优化建议
4. 【可维护性】命名/结构/重复代码改进点
5. 【亮点】值得肯定的写法
6. 【修复示例】关键问题的修复代码

## 示例
输入：「帮我 review 这个 React 组件」（附代码）
输出：总体：逻辑清晰但有 2 处阻塞问题 → 阻塞：useEffect 依赖缺失导致闭包取到旧值；innerHTML 拼接用户输入有 XSS 风险 → 性能：render 内未缓存的 options 映射 → 可维护性：200 建议抽为常量 PAGE_SIZE → 亮点：请求取消的封装规范 → 修复示例：useMemo 缓存 + DOMPurify 净化。

## 注意事项
- 对事不对人，用「建议」「是否可以」的表达
- 阻塞问题必须说明原因和修复建议，不只说「这样不好」
- 优先发现正确性与安全问题，不纠缠风格争论
- 结合团队 lint 与规范文档，不强制个人偏好
- 肯定好的写法，审查也是团队互相学习的过程
