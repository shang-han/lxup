0.2 一机一码防复制原理
核心问题：V1.0 的 U 盘物理分发无法防止复制——用户拷贝 U 盘文件到另一台机器即可使用。
V2.0 解决方案：
  1. 激活码首次激活 → License Server 将码与设备指纹（硬件特征哈希）绑定
  2. 本地 token 用设备指纹派生的密钥加密存储，换机后无法解密
  3. 同一激活码在另一台机器激活时 → 服务器检测到设备指纹不匹配 → 拒绝
  4. 用户换电脑（正当场景）→ 联系客服解绑 → 在新机器上重新激活
  5. 离线使用仍支持 3 天宽限期，但首次激活必须联网

0.3 激活码格式
基础版（1 个岗位）：LXUP-B-8A3F9C
Pro 版（3 个岗位）：  LXUP-P-7D2E1B

前缀标识 license 级别（B=Basic / P=Pro），后 6 位为随机码，总空间 2,176,782,336 种组合。
激活码由 License Server 批量生成并管理。

1. 系统架构设计
采用 **”Tauri v2 主壳 + Python Sidecar + 异构 Agent Worker”** 的三层架构：

- **Tauri v2**：作为桌面应用主壳，承载 Web UI (Vue)，并通过其内置 **Sidecar API** 管理 Python Gateway 子进程。Tauri v2 原生支持 sidecar 进程的生命周期管理（启动/重启/杀进程/stdin-stdout 通信），无需手动 spawn。
- **Python Gateway (Sidecar)**：所有业务逻辑、状态管理、Agent 调度均由 Gateway 承担，编译为独立可执行文件后，通过 Tauri v2 的 `tauri.conf.json` 中 `bundle > externalBin` 声明为 sidecar，随应用自动分发。
- **Agent Worker**：Codex CLI (Node.js sidecar) / OpenClaw / Hermes 由 Gateway 统一管理。

4. 关键业务模块实现指引
4.1 Gateway 生命周期管理（基于 Tauri v2 Sidecar）

Tauri v2 原生提供 Sidecar API（`shell.sidecar`），无需手动管理进程：
- **配置声明**：在 `tauri.conf.json > bundle > externalBin` 中声明 gateway 二进制路径，构建时自动打包到安装目录
- **启动**：Rust 侧通过 `Command::sidecar(“gateway”)` 启动 Python Gateway，传入 `--port` + `--token` 参数
- **进程通信**：Gateway stdout/stderr 通过 Tauri event 桥接到 Web UI；Web UI 可通过 `invoke` → Rust command → 重启 sidecar
- **健康检查**：Tauri Rust 侧定时向 Gateway 发 `/health` 请求；异常时自动重启 sidecar 并通知前端
- **优雅关闭**：应用退出时 Tauri 自动向 sidecar 发 SIGTERM → 等待 5s → SIGKILL，Gateway 需处理 SIGTERM 完成事务提交
- **优势**：无需 .gateway.pid / .gateway.ready 标记文件，Tauri v2 内部管理子进程状态，跨平台行为一致
4.2 会话与记忆持久化
会话表: sessions(id, agent_type, profile_id, messages_json, created_at)

4.5 License 授权校验体系
4.5.1 总体设计
授权采用"激活码 + 设备绑定 + JWT Token"三级架构：

License Server（云端） — 管理激活码的生成、绑定、吊销
Gateway License 服务（本地） — 加密存储 token、离线宽限、设备变更检测
Tauri Rust 侧 — 采集设备指纹，通过 IPC 传给 Web UI，再转发给 Gateway

核心规则：

激活码永久有效，一码绑一机（换机需重新激活）
本地 token 用设备指纹派生的 AES 密钥加密存储（Fernet）
每次启动比对指纹，换机 → 强制在线重验
离线容忍 3 天，超期 → 锁定直到联网

4.5.2 数据库模型
License Server 侧：
激活码表 activation_codes

code_hash: SHA-256(激活码明文)，用于查找
code_prefix: 激活码前 4~8 位明文，便于客服查询
status: active(未使用) / used(已绑定) / revoked(已吊销)
batch_id: 批次号，用于批量管理

设备绑定表 device_bindings

code_id → activation_codes.id（一码一机）
device_fp_hash: SHA-256(设备指纹)
device_name: 用户设置的设备名
last_validated_at: 最后一次在线校验时间

审计日志表 license_audit_logs

action: activate / validate / revoke / reject
记录每次操作的 client_ip，用于风控

Gateway 本地侧：
本地授权表 license_local

token_encrypted: Fernet 加密的 JWT activation_token
device_fp_hash: 用于换机检测
last_online_at: 最后一次在线校验成功时间

4.5.4 启动校验流程
Gateway 启动时执行 validate_on_startup()：

加载本地 license_local 记录
  ├─ 无记录 → NOT_ACTIVATED（未激活）
  └─ 有记录 → 继续

校验设备指纹（当前 fp_hash vs 记录的 fp_hash）
  ├─ 匹配 → 检查离线时长
  │   ├─ < 3 天 → OK（宽限期内）
  │   └─ ≥ 3 天 → 强制在线校验
  │       ├─ 联网成功 → OK，更新 last_online_at
  │       └─ 联网失败 → BLOCKED_OFFLINE（锁定）
  └─ 不匹配 → 设备变更检测
      ├─ 用当前 fp 解密 token 失败 → DEVICE_CHANGED（需重新激活）
      └─ 解密成功 → 在线重验
          ├─ 服务器确认本机 fp 合法 → OK（小硬件变动）
          └─ 服务器拒绝 → DEVICE_CHANGED

4.5.5 在线校验流程
Gateway 定时（可配置）或用户手动触发 verify_online()：

解密本地 token（密钥派生自设备指纹）
  └─ 解密失败 → DEVICE_CHANGED

POST /validate {activation_token, device_fp_hash} → License Server
  ├─ JWT 签名无效 → 拒绝
  ├─ fp_hash 不匹配 → 拒绝
  ├─ 激活码 status=revoked → 吊销，清本地记录
  └─ 通过 → 更新 last_validated_at

4.5.6 加密与安全
设备指纹哈希: SHA-256(raw_fingerprint)
密钥派生: base64(SHA-256("license-key-v1:" + device_fp)) → 32 字节 Fernet key
Token 加密: Fernet(derived_key).encrypt(jwt_token)
JWT 签名: HS256，密钥仅 License Server 持有（LICENSE_JWT_SECRET 环境变量注入）

关键安全属性：
- API Key / Token 绝不落盘明文
- 设备指纹的原始值（硬件序列号等）不出本地，仅传输 hash
- License Server 仅接收 fp_hash，无法逆推原始硬件信息
- Fernet 密钥从设备指纹派生 → 即使数据库文件被盗，在其他机器上也无法解密 token

4.5.7 离线策略总结
场景	状态	用户可见
首次启动，未激活	NOT_ACTIVATED	"请输入激活码"
指纹匹配，在线 < 3 天	OK	"授权正常"
指纹匹配，离线第 N 天 (N<3)	OK	"离线第 N 天，剩余 X 天"
指纹匹配，离线 ≥ 3 天，联网失败	BLOCKED_OFFLINE	"离线超期，请联网验证"
指纹不匹配，重验失败	DEVICE_CHANGED	"检测到硬件变更，请重新激活"
服务器返回吊销	REVOKED	"授权已被吊销"
4.6 端到端加密中继（手机远程访问 — 架构预留）
4.6.1 解决的问题
用户离开桌面端所在网络后（外出、通勤），手机仍能通过中继隧道连接到家中/办公室的 Gateway，继续使用 AI 功能。

4.6.2 技术方案
┌──────────────────────┐         ┌──────────────────────┐
│  手机浏览器            │         │  中继服务器             │
│  (外出时)             │  TLS    │  ¥136/年 × 2          │
│                      │◄───────►│  nginx + WebSocket    │
└──────────────────────┘         └──────────┬───────────┘
                                            │ WSS (TLS)
                                    只转发密文，不解密
                                            │
                              ┌─────────────┴───────────┐
                              │  桌面端 Gateway           │
                              │  (家中/办公室，常驻运行)    │
                              └──────────────────────────┘

流程：
  1. Gateway 启动 → 自动建立 WebSocket TLS 连接到中继服务器
  2. Gateway 上报自己的设备 ID（如 u3827）→ 中继注册该设备在线状态
  3. 手机扫码或访问 https://relay.longxia.cn/u3827
  4. 手机 ↔ 中继 ↔ Gateway，全程 TLS 加密
  5. 中继只做数据包转发，不解密、不存储、不解析

4.6.3 中继服务器的法律角色
- 中继不经手密钥：TLS 握手在手机 ↔ Gateway 之间端到端完成，密钥不经过服务器
- 服务器日志只记录：连接时间、断开时间、传输字节数
- 角色定位：管道工 / 信使，非数据处理者——类似 VPN 服务商的法律地位

4.6.4 稳定性保障
- 心跳机制：每 15 秒 ping，超过 45 秒无响应 → 自动重连
- 桌面端防休眠：Tauri 系统托盘常驻，阻止操作系统休眠
- 双服务器热备：主服务器挂了 → 客户端 3 秒内切换备机
- WSS 走 443 端口：防火墙无法区分 HTTPS 和 WSS，不易被屏蔽
- 回退机制：中继不可用时，Web UI 显示"远程连接不可用"，本地访问不受影响

4.6.5 容量估算
- 每连接占用 10-20KB 内存 → 1GB 内存可支撑 5,000+ 并发连接
- 按 10,000 台激活设备、峰值 20% 同时在线计算 → 2,000 并发
- 两台 ¥136/年的轻量云服务器各承担 1,000 连接，绰绰有余

4.6.6 实施策略
Phase 1 (当前): 架构预留，Gateway 的 WebSocket 路由设计中继扩展点
Phase 2 (后续): 中继服务器（nginx + WebSocket proxy）搭建 + 域名 + TLS 证书
Phase 3 (后续): Gateway 隧道客户端 + 前端远程连接入口 + 扫码连接

5. 打包与分发策略（Tauri v2 + Sidecar）

核心思路：以 Tauri v2 为唯一主程序入口，Python Gateway 和 Codex CLI 均作为 sidecar 子进程随主程序分发。用户安装后只需启动一个 exe，所有子进程由 Tauri 透明管理。

5.1 Tauri v2 Sidecar 配置

在 `src-tauri/tauri.conf.json` 中声明 sidecar 二进制：

```json
{
  "bundle": {
    "externalBin": [
      "binaries/gateway",      // Python Gateway (pyinstaller 产物)
      "binaries/codex-sidecar" // Codex CLI (nexe/pkg 产物)
    ],
    "resources": {
      "default-config.yaml": "config/default-config.yaml",
      "chromium/": "binaries/chromium/"  // Playwright 内置浏览器（可选）
    }
  }
}
```

Tauri v2 构建时会自动将 externalBin 中的可执行文件打包到各平台安装包中，并处理 macOS 签名 / Windows 签名。

5.2 产物结构

```
dist/
├── MyApp.exe                  # Tauri v2 主程序（唯一入口）
│   (内含):
│   ├── Web UI (Vue 3)        # 编译后的静态资源，通过 WebView 渲染
│   ├── gateway.exe           # Python Gateway sidecar（含 OpenClaw/Hermes）
│   ├── codex-sidecar.exe     # Node.js Codex CLI sidecar
│   ├── chromium/             # Playwright 内置 Chromium（可选）
│   └── default-config.yaml   # 默认配置模板
```

用户视角：只有一个 `.exe` / `.app` / `.AppImage`，双击即用。

5.3 运行时 Sidecar 通信流程

```
┌──────────────┐     IPC (invoke)     ┌──────────────┐
│  Web UI (Vue) │ ◄──────────────────► │  Rust (Tauri) │
└──────────────┘                      └──────┬───────┘
                                             │
                                    Sidecar API (spawn/kill/stdio)
                                             │
                              ┌──────────────┴──────────────┐
                              ▼                             ▼
                      ┌─────────────┐              ┌────────────────┐
                      │  gateway.exe │              │ codex-sidecar  │
                      │ (Python)     │              │   (Node.js)    │
                      │ :7889        │              │ stdio JSON-RPC │
                      └──────┬───────┘              └────────────────┘
                             │
                     Stdio JSON-RPC
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ OpenClaw │  │  Hermes  │  │ (其他)   │
        │ (Python) │  │ (Python) │  │          │
        └──────────┘  └──────────┘  └──────────┘
```

Web UI ↔ Gateway 仍走 HTTP/WebSocket（localhost），不经过 Tauri IPC 转发，减少性能开销。Tauri IPC 仅用于生命周期控制（启动/重启/关闭 sidecar）。

5.4 构建流水线

```yaml
# CI/CD 构建步骤
steps:
  # 1. 构建 Python Gateway sidecar
  - name: Build Gateway Sidecar
    run: |
      pip install pyinstaller
      pyinstaller --onefile --hidden-import=openclaw --hidden-import=hermes \
        --add-data "default-config.yaml:." gateway/main.py \
        -o src-tauri/binaries/gateway

  # 2. 构建 Codex CLI sidecar（Node.js → 独立二进制）
  - name: Build Codex Sidecar
    run: |
      cd codex && pnpm build
      npx nexe packages/codex-cli/bin/codex.js -o ../src-tauri/binaries/codex-sidecar

  # 3. Tauri v2 构建（自动捆绑 externalBin）
  - name: Bundle Tauri App
    run: |
      pnpm tauri build --bundles nsis,dmg,appimage
      # Tauri v2 自动将 binaries/ 下的 sidecar 打包进安装包
```

5.5 技术要点

- **跨平台二进制命名**：Tauri v2 支持 `externalBin` 中使用目标平台三元组后缀（如 `gateway-x86_64-pc-windows-msvc.exe`），构建时自动匹配
- **签名**：macOS 需要对 sidecar 二进制单独签名（`codesign`），Windows 需与主程序一起签名
- **首次启动优化**：Chromium 可设为可选资源（首次运行时自动下载），减小安装包体积 ~100MB
- **更新策略**：采用混合更新方案，见 5.6
5.6 混合更新方案（方案 B）
概述：将更新拆成两条正交链路，互不阻塞。链路 1 负责 Tauri Shell + Web UI（低频），链路 2 负责所有 Sidecar 二进制（高频），由 Gateway 自更新服务驱动。

5.6.1 两条更新链路
链路	负责方	更新对象	更新方式
链路 1	Tauri v2 内置 Updater	Rust Shell + Web UI 静态资源	Tauri 原生增量更新
链路 2	Gateway 自更新服务	gateway.exe / codex-sidecar / chromium / skills	独立 manifest + 全量/差分下载

sidecar 更新不需要走 Tauri 签名通道，Tauri 壳更新不需要重启 Agent 运行时。两条链路并行独立工作，用户无感知。

5.6.2 更新 Manifest 格式
Gateway 的更新服务需要一个标准化的 manifest，由 License Server 负责签发：

```yaml
manifest_version: 2
issued_at: “2026-07-15T10:00:00Z”
expires_at: “2026-07-22T10:00:00Z”   # manifest 自身过期时间，防止重放

components:
  gateway:
    current_version: “0.2.0”
    min_version: “0.1.0”            # 低于此版本必须强制更新
    files:
      - platform: windows-x86_64
        url: “https://cdn.example.com/releases/gateway/0.2.0/gateway-win-x64.tar.gz”
        size: 45_000_000
        sha256: “abc123...”
        diff_from:                  # 差分更新（可选）
          - version: “0.1.0”
            url: “https://cdn.example.com/releases/gateway/0.2.0/diff-0.1.0-to-0.2.0-win-x64.patch”
            size: 3_200_000
            sha256: “def456...”
      - platform: macos-arm64
        url: “https://cdn.example.com/releases/gateway/0.2.0/gateway-mac-arm64.tar.gz”
        ...

  codex-sidecar:
    current_version: “1.5.0”
    min_version: “1.0.0”
    files:
      - platform: windows-x86_64
        url: “https://cdn.example.com/releases/codex/1.5.0/codex-sidecar-win-x64.exe”
        ...

  chromium:
    current_version: “128.0.6613.0”
    optional: true                   # 可选组件，缺失不影响核心功能
    files:
      - platform: windows-x86_64
        url: “https://cdn.example.com/releases/chromium/128.0.6613.0/chromium-win-x64.tar.gz”
        size: 150_000_000
        ...

  skills:
    index_url: “https://cdn.example.com/skills/index.json”
    # index.json: [{name, version, url, sha256, dependencies}, ...]

# Ed25519 签名 — License Server 私钥签发，客户端内置公钥验证
signature: “base64-encoded-ed25519-signature”
```

5.6.3 Gateway 自更新服务流程
┌─────────────────────────────────────────────────────┐
│                  Gateway Updater                     │
│                                                      │
│  1. 定时检查 (APScheduler)                            │
│     ├── 每 N 小时向 License Server 拉取 manifest       │
│     ├── 对比本地版本 (versions.json)                   │
│     └── 有更新 → WebSocket 推送事件给前端              │
│                                                      │
│  2. 下载阶段                                          │
│     ├── 优先尝试 diff 差分 (体积小)                    │
│     ├── diff 失败/不存在 → 退化为全量下载              │
│     ├── 断点续传 (Range 请求 + 本地 .part 临时文件)     │
│     └── 下载到 {workspace}/.updates/ 临时目录          │
│                                                      │
│  3. 校验阶段                                          │
│     ├── SHA-256 哈希校验 (防损坏)                      │
│     ├── Ed25519 签名校验 (防篡改)                      │
│     └── 版本兼容性检查 (不低于 min_version)             │
│                                                      │
│  4. 应用阶段 (原子替换)                                │
│     ├── 写入 .updates/staging/ 目录                   │
│     ├── 当前运行文件 → .updates/backup/ (回滚用)       │
│     ├── 通知 Tauri Rust 侧重启 sidecar                │
│     └── Tauri 执行 rename staging → 正式目录           │
│                                                      │
│  5. 回滚阶段                                          │
│     ├── 新版本启动失败 → 自动回滚到 backup             │
│     ├── 用户手动回滚 (前端按钮)                        │
│     └── 保留最近 3 个版本的 backup                     │
└─────────────────────────────────────────────────────┘

关键设计决策 — Windows exe 运行中无法替换自身：
Gateway 只负责 下载 + 校验 + 写入 staging 目录 + 写 .pending_update 标记文件，然后通知 Tauri Rust 侧：
  {“method”: “update.ready”, “params”: {“component”: “gateway”, “version”: “0.2.0”}}

Tauri Rust 收到通知后：
  1. 向 Gateway 发 /shutdown → 等待优雅退出
  2. 执行 rename（staging → 正式路径）
  3. 重新 spawn Gateway sidecar
  4. Gateway 启动后检查 backup → 3 次启动失败 → 自动 rollback

5.6.4 与 License 体系的集成点
Gateway 拉取 manifest 时带上 activation_token，License Server 根据 license 级别返回裁剪后的 manifest：

```
POST /api/update/manifest
Body: {
  “activation_token”: “eyJ...”,
  “device_fp_hash”: “sha256...”,
  “current_versions”: {
    “gateway”: “0.1.0”,
    “codex-sidecar”: “1.4.0”,
    “chromium”: null
  },
  “platform”: “windows-x86_64”
}
Response: ← 按 license 级别裁剪过的 manifest
```

三种绑定策略（以开关形式控制）：
- 不绑定：所有已激活用户均可更新（常规 bugfix 和安全补丁）
- 按 License 级别：basic 用户只能更新到 x.y.z，pro 用户能更新到 x.y.z+1（功能分级）
- 按 License 有效期：License 过期后仍可使用当前版本，但不能获取更新（SaaS 订阅模式）

5.6.5 本地版本信息存储
Gateway 本地维护 versions.json（打包时写入，随 sidecar 一起分发）：

```json
{
  “gateway”: {
    “version”: “0.1.0”,
    “build_time”: “2026-07-14T12:00:00Z”,
```

5.6.6 Skills 技能包独立更新
▎设计原则
- 增量更新：只下载变化的文件，不重下全量 50 岗位 + 100 技能
- 版本号驱动：V1.0 → V1.1 → V1.2 链式升级，自动计算最短升级路径
- 断点续传：下载中断后从断点继续
- 回滚安全：更新前自动备份旧版本，失败自动回滚
- 每月新增 5-10 个通用技能 → 主版本号升级
- 每季度优化岗位包 → 次版本号升级

▎文件级版本标记
每个技能文件头部带版本号注释：

// @version 1.2.0
// @updated 2026-07-15
// @min_gateway 0.1.0

skill.yaml 中的版本定义：

name: browser-automation
version: "1.2.0"
min_gateway: "0.1.0"
dependencies:
  - name: file-system
    version: ">=1.0.0"
files:
  - path: tools.py
    sha256: "abc123..."
    version: "1.2.0"

▎增量包格式
远程增量包为 .zip，仅含变更文件 + 更新清单：

V1.2.zip
  ├── manifest.json           # 本次更新的变更清单
  │   {
  │     "from_version": "1.1.0",
  │     "to_version": "1.2.0",
  │     "changes": [
  │       "新增3个通用技能：邮件助手、会议纪要、日程管理",
  │       "优化房产中介话术模板",
  │       "修复文件系统技能的路径转义问题"
  │     ],
  │     "files": [
  │       {"path": "房产中介/system_prompt.md", "sha256": "...", "action": "update"},
  │       {"path": "邮件助手/", "sha256": null, "action": "create"}
  │     ],
  │     "total_size_kb": 156
  │   }
  └── 房产中介/
      └── system_prompt.md    # 变更的文件（按原目录结构）

▎更新 API
GET /api/skills/check-update?current_version=V1.1
  → {
      "latest": "V1.2",
      "update_chain": ["V1.1→V1.2"],   // 链式升级路径，跳版本时可能多步
      "total_size_kb": 156,
      "changes": ["新增3个通用技能", "优化房产中介话术"]
    }

GET /api/skills/download/V1.2
  → 返回增量包 .zip

▎本地更新流程
1. 用户点「检查更新」/ 定时自动检查
2. Gateway 向技能更新服务器发送当前版本号
3. 服务器计算最短升级路径（如 V1.0 → V1.1 → V1.2，或 V1.0 → V2.0 直接跳）
4. 返回更新链 + changelog + 文件大小
5. 用户确认 → Gateway 依次下载每个版本的增量包
6. 每步：下载增量包 → SHA-256 校验 → 备份旧文件 → 解压覆盖 → 验证完整性
7. 全部完成 → 更新本地 index.json → 重载 Agent 工具注册表
8. 任一步失败 → 从备份恢复 → 报告更新失败

5.6.8 CI/CD 构建 + 发布流水线
在原构建步骤上增加 manifest 生成和签名：

```yaml
steps:
  # ...（原有构建步骤保持不变）...

  # 4. 生成更新 manifest
  - name: Generate Update Manifest
    run: |
      python scripts/generate-manifest.py \
        --gateway-version ${{ github.ref_name }} \
        --codex-version $(cat codex/VERSION) \
        --artifacts-dir ./dist/ \
        --output manifest.yaml

  # 5. Ed25519 签名 manifest（私钥存储在 CI secret）
  - name: Sign Manifest
    run: |
      python scripts/sign-manifest.py \
        --input manifest.yaml \
        --private-key ${{ secrets.UPDATE_SIGNING_KEY }}

  # 6. 上传到 CDN
  - name: Upload to CDN
    run: |
      aws s3 sync ./dist/releases/ s3://cdn.example.com/releases/ \
        --cache-control “public, max-age=31536000, immutable”
      aws s3 cp manifest.yaml s3://cdn.example.com/releases/manifest.yaml \
        --cache-control “public, max-age=300”  # manifest 缓存 5 分钟

  # 7. 通知 License Server 更新 manifest
  - name: Notify License Server
    run: |
      curl -X POST https://license.example.com/admin/update-manifest \
        -H “Authorization: Bearer ${{ secrets.ADMIN_TOKEN }}” \
        -H “Content-Type: application/json” \
        -d '{“manifest_url”:  “https://cdn.example.com/releases/manifest.yaml”}'
```

5.6.9 安全措施清单
层面	措施
传输	HTTPS only, TLS 1.2+
完整性	SHA-256 哈希校验每个文件
防篡改	Ed25519 签名 manifest，公钥硬编码在 Gateway 中
防重放	manifest 自带 issued_at / expirses_at，过期拒绝
防降级	拒绝安装版本号低于当前的 manifest（除非显式 rollback 且签名有效）
原子性	staging → rename，绝不直接覆盖运行中文件
回滚	保留最近 3 个版本 backup，启动失败自动回滚
权限	更新文件写入 {workspace}/.updates/，不涉及系统目录（Windows 不需要 admin）

5.6.10 实施优先级
Phase 1 (基础):   Gateway 更新服务 + manifest 拉取 + 版本对比 + 前端通知
Phase 2 (下载):   全量下载 + 断点续传 + SHA-256 校验
Phase 3 (应用):   Tauri Rust 侧 rename + 重启 sidecar + 回滚机制
Phase 4 (高级):   Ed25519 签名校验 + 差分更新 + License 绑定
Phase 5 (技能):   Skills 市场独立更新系统

5.7 安全与合规
Gateway 监听 127.0.0.1 仅限本地访问，禁止绑定 0.0.0.0
Tauri v2 启动 sidecar 时通过 `--token` 传入随机生成的认证 Token，Gateway 校验所有 HTTP/WebSocket 请求的 `Authorization: Bearer <token>` 头
API Key 加密存储于系统 Keychain / Credential Manager，不落盘明文
Codex CLI 沙箱默认启用，UI 中”无视风险模式”需二次确认并记录审计日志
6. 用户完整体验流程
6.1 第一次使用
① 下载安装包（.exe / .dmg / .AppImage），双击安装
② 启动 → 弹出「龙虾 AI」桌面窗口（WebView）
③ 看到激活界面 → 输入购买的激活码（如 LXUP-P-7D2E1B）
④ 激活成功 → 进入岗位选择页面（50 个岗位卡片，含图标+简介）
⑤ 基础版勾选 1 个 / Pro 版勾选 3 个岗位 → 点「确认」
⑥ 其余岗位卡片变灰，显示「升级 Pro 解锁」
⑦ 填写 API Key（DeepSeek / 百炼 / 通义千问）→ 完成配置
⑧ 进入 Dashboard → 开始与 AI 助手对话

6.2 每一天
① 开机 → 启动龙虾 AI（随系统自启动，系统托盘常驻）
② 坐在电脑前 → 桌面窗口直接聊
③ 去茶水间 / 下楼吃饭 → 手机打开收藏夹 URL → 自动走中继隧道（如果已部署）→ 继续聊
④ 本地网络下（同一 WiFi）→ 手机浏览器直接访问桌面端 IP:端口，不走中继，延迟更低
⑤ 回家 → 电脑还开着 → 被窝里继续聊（经中继）

6.3 每月一次
① 点「检查更新」
② 显示「本月新增 8 个通用技能，3 个岗位技能优化」→ 点「立即更新」
③ 下载增量包 ~200KB → SHA-256 校验 → 自动覆盖 → 重启 Gateway
④ 技能库扩充完毕 → 继续聊——新技能已生效

6.4 用户换电脑
① 在新电脑安装龙虾 AI
② 输入原激活码 → 提示「此激活码已绑定其他设备，是否申请解绑？」
